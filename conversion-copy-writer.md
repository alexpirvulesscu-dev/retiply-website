# Conversion Copy Writer

A skill for producing website copy that actually converts. Every line delivered through this skill must embody the 10 principles below — not describe them, not grade against them, but *be* them.

## When to use this skill

- User asks to write or rewrite website copy (headline, subhead, hero, CTA, PS, full page)
- User gives a draft and says "make this better" / "sharpen this" / "fix this"
- User is iterating on Retiply, CASS MUC, or any other venture's marketing copy
- User gives constraints like "max 6-7 words" — treat those as hard rules, not suggestions

Do **not** use this skill for:
- Visual design, layout, or creative direction
- Brand identity or logo work
- Long-form content (blogs, articles, whitepapers) — this is for conversion copy only

## The 10 principles (what every line must do)

Every piece of copy delivered should pass as many of these as the format allows. A headline can't do all 10, but the *page* should. Before delivering, check the output against this list.

### 1. Headlines carry 80% of the weight
Most visitors only read the headline. It must grab them by the throat. Generic statements any competitor could make are failure. Specific, visceral, curiosity-inducing, or status-triggering wins. If the headline could sit on a competitor's site unchanged, rewrite it.

### 2. Third-grade reading level
Short words. Short sentences. No jargon. If a sentence needs a second read, cut it. Test: would a distracted, rushed reader get it in one glance?

### 3. Specific moments, not abstractions
Don't write "manual tasks draining your business." Write "your ops manager copying WhatsApp messages into a spreadsheet at 10pm." Paint one concrete scene from the customer's actual day. Name the tool, the time, the person, the frustration. Vague = dead.

### 4. Avatar callout + anti-avatar
Say exactly who it's for. Then say who it's *not* for. "If you're doing under £500k, this isn't for you" builds more authority than any credentials list. Explicit exclusion = self-selection = better leads.

### 5. Damaging admissions
Own a flaw before claiming a benefit. "We're not the cheapest." "We only take 3 clients a quarter." "Our onboarding is slow because we do it right." Admitting a weakness kills skepticism and makes everything that follows believable.

### 6. Proof, not promises
Specific numbers. Named clients. Dated results. "Saved [company] 18 hours a week in 6 weeks" beats "save time." If there's no real proof yet, **say so honestly** — don't invent anonymous testimonials. Fabricated social proof destroys trust faster than having none.

### 7. Reason why
Every offer needs a "because." Free call *because* we only take 3 new clients a quarter. Discount *because* it's launch month. Limited spots *because* each engagement needs 40 hours of senior time. No reason = no urgency = no action.

### 8. Tie benefits to status
Humans compete. Frame outcomes as "how will your board / peers / team / competitors see you after this?" Functional benefits (save time, reduce chaos) are table stakes — pair them with social consequence ("the partner who finally stopped firefighting").

### 9. PS / closing recap
The last line of a page is the second-most-read after the headline. Use it to restate the core promise, repeat the anti-avatar, or hit one last status note. A weak fade-out ("thanks for visiting") wastes the highest-attention moment on the page.

### 10. Stupid-simple CTAs
Tell the reader: what to click, what happens next, what they'll get. "Step 1: click here. Step 2: pick a 30-min slot. Step 3: we send a 1-page prep doc before the call." Never assume the reader knows what happens after the button.

## Workflow

1. **Get the inputs.** Before writing, confirm:
   - What page/section is this for? (hero, about, pricing, CTA block, full page)
   - What's the business, ICP, and offer?
   - Any constraints? (word count, tone, words to include/avoid, specific claim to lead with)
   - What proof exists? (real numbers, named clients, case studies — or honestly none yet)
   - If anything critical is missing, ask **before** writing. Don't guess on ICP or invent proof.

2. **Draft with the principles baked in.** Don't write "generic-ish" copy and then try to sharpen it. Write each line already passing principle 3 (specificity) and principle 1 (throat-grab) from the first draft.

3. **Offer variants for high-stakes copy.** For headlines, taglines, and primary CTAs, give **2–3 options** with different angles (e.g. pain-led, outcome-led, status-led). One line under each explaining the angle. Let the user pick.

4. **Deliver exact copy, not theory.** Ana is non-technical and wants words she can paste. Never say "you could talk about your experience here" — write the actual sentence.

## Hard rules for the writing itself

**Banned phrases** (unless the user explicitly asks for them):
- "Streamline your operations"
- "Unlock your potential"
- "Powered by AI" / "AI-powered"
- "Take your business to the next level"
- "Revolutionary" / "cutting-edge" / "best-in-class"
- "Seamless" / "frictionless" / "effortless"
- Any phrase that could appear verbatim on 1,000 other SaaS homepages

**Length defaults** (override if the user specifies):
- Headline: ≤ 10 words, ideally 5–8
- Subhead: ≤ 20 words, one sentence
- Body paragraph: ≤ 3 short sentences
- CTA button: 2–5 words
- PS: 1–2 sentences

**Specificity test** — before delivering any line, ask: *can I picture the moment?* If the answer is no, rewrite it. "Repetitive work" fails. "Pulling invoice data out of emails into a spreadsheet every Friday" passes.

**Proof honesty** — if the user has real numbers or named clients, use them. If they don't, **do not fabricate**. Instead, write copy that leans on what's true (founder's operational experience, specific process, explicit anti-avatar) and flag to the user: "add a real case study here once you have one — for now this section leans on founder credibility."

**Status layering** — for every functional benefit you write, ask whether you can add one status line. "Stop firefighting at 9pm" (functional) + "be the operator your team stops paging on weekends" (status).

## Output format

**For single lines (headlines, taglines, CTAs):**

```
Option A — [angle, e.g. "pain-led"]
[the actual line]

Option B — [angle, e.g. "outcome-led"]
[the actual line]

Option C — [angle, e.g. "status-led"]
[the actual line]
```

**For full sections or pages:**
Deliver the copy in the exact order it appears on the page, labeled by section (Hero / Subhead / Body / CTA / PS). No explanation between sections unless the user asks. If something leans on proof that doesn't exist yet, add a single bracketed note: `[replace with real case study number once available]`.

**Never deliver:**
- Principles-explained-back-to-the-user ("this headline uses specificity because…")
- Multiple paragraphs of reasoning before the copy
- Copy with a placeholder like "[your value prop here]" — write the actual value prop
